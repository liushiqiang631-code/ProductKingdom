# pm

## 一、背景与目标

### 1.1 背景
贪吃蛇作为一款经典益智游戏，深受各年龄层玩家喜爱。为满足先帝（用户）对娱乐与怀旧的需求，同时为后续扩展其他经典游戏积累开发经验，特此提出本产品需求。

### 1.2 核心问题
市面上贪吃蛇应用虽多，但高质量、界面简洁、操作流畅、适配多种设备的版本仍属稀缺。现有部分产品存在广告过多、游戏体验差、缺乏本地化设置等问题。

### 1.3 预期目标
- 开发一款具备完整游戏逻辑、操作直观、视觉清爽的贪吃蛇游戏。
- 支持单人模式，提供基础分数显示、游戏暂停/继续功能。
- 确保在主流移动端（iOS/Android）及PC浏览器上流畅运行，延迟低于100ms。
- 无内购、无强制广告，保证纯粹的游戏体验。

---

## 二、用户故事

1. **作为玩家**，我希望通过方向键或滑动控制蛇的移动，以便灵活操控蛇吃食物。
2. **作为玩家**，我希望在游戏过程中看到实时计分与当前长度，以便了解自己的成绩。
3. **作为玩家**，我希望在蛇撞墙或撞身后弹出游戏结束提示并显示最终得分，以便开始新一轮游戏。
4. **作为新玩家**，我希望在游戏开始前看到简单的操作说明，以便快速上手。
5. **作为中重度玩家**，我希望游戏提供速度调节或难度选择，以便挑战更高水平。

## 四、非功能需求

1. **性能要求**
   - 游戏主循环帧率不低于30FPS，推荐60FPS。
   - 从用户操作到蛇头方向变化响应时间≤50ms。
   - 内存占用≤50MB，运行稳定无卡顿。

2. **兼容性要求**
   - 支持主流浏览器最新版本（Chrome、Firefox、Safari、Edge）。
   - 移动端适配：iOS 12+、Android 8+，支持触控滑动和横竖屏。
   - 屏幕分辨率自适应，从320px宽到1920px宽均能完整显示。

3. **可用性要求**
   - 控件标识清晰，重新开始按钮距离游戏结束提示不超过50px。
   - 暂停状态下游戏画面冻结，保证玩家可随时恢复。
   - 游戏过程中不会弹出任何广告或干扰性弹窗。

4. **安全性要求**
   - 无需用户注册登录，不收集任何个人隐私数据。
   - 本地存储仅用于记录最高分，不涉及网络传输。

## 一、技术选型

| 类别 | 技术选择 | 选择理由 |
|------|---------|---------|
| **前端框架/库** | 原生 HTML5 + Canvas 2D | 游戏逻辑简单，无需引入重型框架；Canvas 提供硬件加速渲染，满足60FPS性能需求；无外部依赖，利于跨平台兼容。 |
| **交互控制** | 键盘事件 + 触控滑动事件（`touchstart`/`touchmove`/`touchend`） | 原生事件支持度广，无需额外库；触控滑动逻辑可封装为独立模块。 |
| **音效** | Web Audio API（生成简单合成音效） | 无需加载外部音频文件，减小体积；支持所有现代浏览器。 |
| **存储** | `localStorage`（最高分记录） | 纯前端本地持久化，无需后端，符合无隐私收集要求。 |
| **构建工具** | 裸项目（无构建）或 Vite（可选） | 若需压缩与兼容性，可用 Vite 快速打包；但纯静态单页游戏可直接用 `<script>` 标签。 |
| **测试** | 无（本方案不涉及，但可添加纯逻辑单元测试） | 游戏逻辑简单，采用手动测试与验收标准对照。 |

**补充说明：**  
- 无需后端服务、数据库、中间件，完全属于客户端应用。  
- 未来若需扩展多人或排行榜，再引入 WebSocket + 简单后端（如 Node.js + Redis）。

## 三、数据流图（文字描述）

### 核心业务流程（以玩家操控蛇开始）

1. **初始化阶段**  
   - 页面加载 → `TutorialOverlay` 根据 `localStorage` 判断是否为首次访问，若是则显示操作指引。  
   - 玩家点击“开始游戏”或关闭指引 → 初始化 `GameCore`（蛇初始位置、长度3、方向右），启动游戏循环（`requestAnimationFrame`）。

2. **玩家输入**  
   - 键盘方向键按下 → `InputManager` 捕获事件，过滤反向180°（若当前方向为右，则左键无效），向 `GameCore` 发送 `changeDirection` 指令。  
   - 移动端触摸滑动 → `InputManager` 计算滑动向量（水平/垂直），解析为方向，同样发送指令。

3. **游戏主循环（每帧）**  
   - `GameCore.update(deltaTime)`：  
     - 检查游戏状态是否为 `playing`，是则进行下一步。  
     - 根据当前方向移动蛇头到下一格，身体跟随。  
     - 判断新蛇头位置：  
       - 若吃到食物 → 蛇身不缩短（长度+1），得分+1，触发 `eat` 事件 → `SoundManager` 播放吃食物音效；`StorageManager` 判断是否更新最高分；重新生成食物。  
       - 若撞墙或撞身 → 游戏状态置为 `gameover`，触发 `die` 事件 → `SoundManager` 播放死亡音效；`StorageManager` 保存最高分（若破纪录）。  
       - 若移动合法 → 更新蛇数据。  
   - `Renderer.render(gameState)`：  
     - 清除画布。  
     - 绘制网格（可选）、蛇身（不同颜色区分头）、食物。  
     - 若游戏结束，绘制半透明遮罩 + 最终得分。  
     - 若暂停，绘制“已暂停”文字。

4. **UI更新**  
   - 计分板由 `GameCore` 的 `score` 和 `length` 驱动，直接更新DOM（或通过事件通知UI模块刷新）。

5. **重新开始**  
   - 点击“重新开始”按钮 → 重置 `GameCore` 状态，清除画布，重新开始游戏循环。

### 数据流向图（文字描述）

```
用户方向输入 → InputManager → GameCore.update()
                                    ↓
                             GameCore 内部处理逻辑（碰撞、得分）
                                    ↓
                         触发生成渲染指令（Renderer.render()）
                                    ↓
                           更新计分DOM + 音效播放 + 存储
```

## 四、接口设计概要

由于无后端，所有接口均为前端内部逻辑方法。但为便于代码组织，可定义以下主要 API（伪代码接口）：

### 4.1 GameCore 接口
```javascript
class GameCore {
  constructor(gridWidth, gridHeight, speed)   // 初始化游戏区域尺寸、速度（帧间隔毫秒）
  start()                                      // 启动游戏循环
  pause()                                      // 暂停
  resume()                                     // 继续
  togglePause()                                // 切换暂停/继续
  changeDirection(direction)                   // 改变蛇移动方向（'up','down','left','right'）
  reset()                                      // 重置游戏（重新开始）
  getState()                                   // 返回当前游戏状态：{ status, score, length, snake, food, ... }
}
```

### 4.2 Renderer 接口
```javascript
class Renderer {
  constructor(canvas)                          // 绑定canvas元素
  render(gameState)                            // 根据游戏状态绘制画面
  resize()                                     // 响应窗口尺寸变化，自适应缩放
}
```

### 4.3 InputManager 接口
```javascript
class InputManager {
  constructor(gameCore)                        // 绑定核心逻辑实例
  enable()                                     // 开始监听事件
  disable()                                    // 停止监听（如游戏结束时）
  // 内部处理键盘/touch事件
}
```

### 4.4 存储管理器接口
```javascript
class StorageManager {
  static getHighScore()                        // 获取最高分
  static setHighScore(score)                   // 保存最高分（仅当新分更高时）
}
```

### 4.5 外部事件（自定义事件）
- `game:eat` — 吃食物时触发，传递当前得分  
- `game:die` — 游戏结束时触发  
- `game:scorechange` — 得分变化时触发  
- `game:pausechange` — 暂停状态变化时触发

## 五、部署架构

### 5.1 部署方案
采用 **纯静态单页应用（SPA）** 部署，无服务器端。所有资源（HTML + CSS + JavaScript + 音效合成代码）打包为一个文件或少量文件。

- **环境要求**：仅需一种静态文件服务器。  
  - 开发环境：`http-server`、`live-server` 或 `Vite dev server`。  
  - 生产环境：NGINX、Apache 或云对象存储（如阿里云OSS、AWS S3 + CloudFront）。  

- **跨设备访问**：通过 URL 直接访问，无需安装，适合分享。

### 5.2 微服务/单体说明
本项目为纯前端应用，不涉及微服务。未来若增加联机排行榜或多人模式，可增加一个简单的RESTful后端（Node.js + Express + MongoDB）部署在云服务器，前端通过 HTTP API 通信。

### 5.3 网络要求
- 无网络需求（音效和存储均在本地）。  
- 最高分通过 `localStorage` 存储，不依赖网络。

### 5.4 性能优化
- 使用 `requestAnimationFrame` 驱动渲染循环，自动适配显示器刷新率（60FPS或更高）。  
- 渲染只更新变化的格子（脏矩形优化）可进一步优化，但当前需求下全量重绘即可满足性能。  
- Canvas 缩放采用 CSS 结合 devicePixelRatio 处理，确保高清屏不模糊。

## 六、风险与权衡

### 6.1 已知技术风险

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| 移动端触摸滑动与页面滚动冲突 | 玩家滑动时可能触发页面上下滚动，导致操作失灵 | 在触摸事件中调用 `preventDefault()`，并在CSS中禁止触摸滚动。 |
| 不同浏览器对 Canvas 文字/绘制差异 | UI字体、颜色显示不一致 | 使用 Web 安全字体，明确指定 CSS `font-family`；测试主流浏览器。 |
| 触控方向判定的模糊性（斜向滑动） | 玩家手指斜滑，算法可能误判为水平或垂直 | 采用阈值判定：取水平/垂直方向中位移绝对值较大的方向。 |
| localStorage 在隐私模式下不可用 | 最高分无法保存但游戏应正常运行 | 尝试写入 `localStorage`，捕获异常后忽略存储功能，仍可正常游戏。 |
| 游戏帧率意外下降（如老旧设备） | 蛇移动不均匀，体验差 | 使用基于时间步长的移动逻辑（固定时间增量，而非帧同步），确保在不同帧率下速度一致。 |
| 首次打开时操作指引的 Cookie/存储判断依赖 localStorage | 若用户清除数据，指引会再次显示 | 属于可接受行为，无需额外处理。 |

### 6.2 已知限制
- **无联机功能**：当前设计纯本地，无法与好友对战或全球排行榜（但需求未要求）。  
- **无设置记忆**：难度选择、主题等未持久化（P2功能，可后续扩展）。  
- **音效容量受限**：使用 Web Audio 合成音效，音质简单但体积极小。若需要复杂音效，可考虑 base64 嵌入短音频，但会增加包体积。  

### 6.3 权衡决策
1. **Canvas vs DOM**  
   - 采用 Canvas 放弃 DOM 方式，因为游戏需要频繁重绘（每帧），Canvas 性能更优；DOM 布局不适合动态网格对象。  
2. **纯原生 vs Phaser 等游戏引擎**  
   - 游戏引擎（如 Phaser）提供物理、动画等高级功能，但贪吃蛇逻辑简单且引擎体积大（几百KB）。选择原生开发使加载更快，且方便后续定制。  
3. **单文件 vs 模块化**  
   - 采用 ES6 模块（`type="module"`）组织代码，利于维护。部署时可打包为单文件减小请求量。  
4. **无外部依赖**：完全避免第三方库，确保无安全漏洞和兼容性问题，符合“无广告、无收集”的皇帝旨意。

## 一、数据模型设计

### 1.1 蛇（Snake）
| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `segments` | `Array<{x: number, y: number}>` | 长度 ≥ 1，x, y 为非负整数 | 蛇身坐标列表，索引0为蛇头。 |
| `direction` | `Direction` | 枚举 `'UP' 'DOWN' 'LEFT' 'RIGHT'` | 当前移动方向。 |
| `alive` | `boolean` | 初始为 `true` | 标记蛇是否存活。 |

### 1.2 食物（Food）
| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|

## 二、API 接口详细设计

鉴于无HTTP远程调用，以下为此“后端”模块向外暴露的**函数接口**，由渲染层或输入管理器调用。

### 2.1 游戏生命周期接口

| 方法 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `init(options?)` | `{gridWidth?:20, gridHeight?:20, speed?:200}` | `GameState` | 初始化游戏，生成初始蛇（中心位置，长度3，朝右）和第一个食物，状态设为 `IDLE`。 |

| `pause()` | 无 | `GameState` | 切换暂停，`PLAYING`→`PAUSED`；`PAUSED`→`PLAYING`。 |
| `end()` | 无 | `GameState` | 游戏结束，状态改为 `GAME_OVER`，更新最高分到 localStorage。 |
| `tick()` | 无 | `GameState` | 执行一步移动，内部调用 `move()`、`checkCollision()`、`growIfAte()`，返回最新状态。 |

|------|------|--------|------|
| `changeDirection(newDir)` | `Direction` | `boolean` | 尝试改变蛇方向。若newDir为当前方向的逆方向或与游戏规则冲突则返回false；否则更新direction并返回true。 |
| `handleInput(eventType, data)` | `eventType`，`data` | `boolean` | 统一输入入口，根据事件类型（keyboard/touch）解析方向并调用`changeDirection`。 |

| 方法 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `getState()` | 无 | `GameState` | 返回当前游戏状态的深拷贝。 |

| 方法 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `generateFood()` | 无 | `Food` | 在空白格中随机生成一个食物。 |

## 三、核心业务逻辑

### 3.1 游戏主循环（伪代码）

```
while (gameState.status == 'PLAYING') {
    // 1. 输入缓冲区处理（若存在待处理方向）
    if (pendingDirection != null) {
        changeDirection(pendingDirection);
        pendingDirection = null;
    }

    // 2. 尝试移动蛇头一步（预判）
    newHead = getNextHeadPosition(gameState.snake.segments[0], gameState.snake.direction)
    collisionType = checkCollision(newHead, gameState.snake.segments, gameState.gridSize)

    if (collisionType != 'none') {
        // 3. 碰撞 → 游戏结束
        end()
        break
    }

    // 4. 判断是否吃到食物
    if (newHead.equals(gameState.food.position)) {
        // 吃食物：增长，不删除尾部
        gameState.snake.segments = [newHead] + gameState.snake.segments  // 实际为unshift
        gameState.score += 1
        gameState.food = generateFood() // 生成新食物
    } else {
        // 普通移动：加头去尾
        gameState.snake.segments = [newHead] + gameState.snake.segments.slice(0, -1)
    }

    // 5. 通知渲染层重绘（通过事件/回调）
    render(gameState)

    // 6. 等待 (speed) 毫秒
    sleep(gameState.speed)
}
```

### 3.2 方向控制与反向拦截

```
function changeDirection(newDir):
    if newDir 是当前方向的逆方向（如当前UP，新DOWN）：
        return false   // 拒绝反向
    否则：
        gameState.snake.direction = newDir
        return true
```

### 3.3 食物生成算法

```
function generateFood():
    emptyCells = 所有坐标 (0≤x<gridWidth, 0≤y<gridHeight) 中除去蛇身占用的格子
    if emptyCells 为空：
        // 全屏占满，玩家胜利（可扩展为胜利状态）
        return null   // 或触发胜利处理
    randomIndex = random(0, emptyCells.length - 1)
    return { position: emptyCells[randomIndex], type: 'normal' }
```

### 3.4 暂停/继续

由渲染层捕获空格键或点击暂停按钮，调用 `pause()` 方法，其内部仅切换状态 `PLAYING ↔ PAUSED`。主循环在 `PAUSED` 状态时阻塞，不执行 `tick()`。

## 五、错误处理与安全

### 5.1 异常处理策略

- **输入异常**：`changeDirection` 在收到非法方向字符串时返回 `false`，不改变状态。
- **边界溢出**：游戏区域坐标严格限定在 `[0, gridWidth-1] × [0, gridHeight-1]`，所有坐标运算后需进行范围校验。
- **食物生成失败**：当蛇身占满所有格子时，`generateFood` 返回 null，应视作玩家获胜（可显示“你赢了”并结束游戏）。当前版本可简化为游戏继续无食物（但不会发生），或默认胜利。
- **localStorage 异常**：使用 `try-catch` 包裹读写操作，若存储空间不足或禁用，最高分功能降级为不保存（不影响游戏体验）。

### 5.2 鉴权方案

- 本游戏无用户身份概念，无需鉴权。
- 所有游戏数据完全在客户端内存中，无人为篡改风险。

### 5.3 数据校验规则

| 数据项 | 校验规则 |
|--------|----------|
| 方向值 | 必须是四种枚举之一，忽略大小写。 |
| 坐标 | 非负整数，且位于网格范围内。 |
| 蛇身长度 | 最小1（蛇头），游戏开始为3；不可手动设置。 |
| 食物位置 | 必须为空闲格子，且不与蛇身重叠。 |
| 速度 | 仅允许三个预设值，若外部传入非法值，回退至普通速度200ms。 |
| 游戏状态转换 | 严格遵循状态机：IDLE→PLAYING→PAUSED↔PLAYING→GAME_OVER→（重新开始）→PLAYING。 |

### 5.4 安全建议

- 所有用户输入（键盘、触摸）均通过 `InputManager` 集中解析，不直接操作游戏核心，防止恶意注入。
- 游戏内与外部环境（如URL参数）无交互，无XSS、CSRF风险。
- 不收集任何个人信息，符合隐私法规。

## 三、状态管理方案

### 3.1 状态管理选型

由于游戏逻辑简单、无复杂组件层级交互，且采用纯原生 JavaScript + Canvas，**不引入外部状态管理库**。采用 **单例模式 + 观察者（EventBus）** 实现轻量级状态管理。

- **核心状态对象**：`GameCore` 内部维护完整的 `GameState`（包括蛇、食物、得分、状态机等）。该对象是唯一的状态源。
- **状态变更方式**：所有状态修改均通过 `GameCore` 的公开方法（`init`, `start`, `pause`, `tick`, `changeDirection` 等）完成，禁止外部直接赋值。
- **状态通知机制**：`GameCore` 在每次状态变更后（如 `tick` 执行后、`pause` 之后）触发自定义事件，UI 组件监听并拉取最新状态快照进行渲染。

### 3.2 核心状态结构

```javascript
// 状态快照（每次渲染时从 GameCore 获取）
const gameState = {
  status: 'IDLE' | 'PLAYING' | 'PAUSED' | 'GAME_OVER',
  snake: {
    segments: [{x, y}, ...],  // 蛇身坐标，索引0为蛇头
    direction: 'RIGHT',
    alive: true
  },
  food: {
    position: {x, y},
    type: 'normal'
  },
  score: 0,
  length: 3,
  gridSize: { width: 20, height: 20 },
  speed: 200, // 毫秒/步
  maxScore: 0 // 从 localStorage 读取
};
```

### 3.3 事件流

- `GameCore` 内部持有一个 `EventBus`（简单实现，如 `{on, emit, off}`）。
- 关键事件：
  - `'tick'` — 每执行完一次移动后触发，参数为最新状态快照。
  - `'gameover'` — 游戏结束时触发，携带最终得分。
  - `'pause'` — 暂停/继续状态变更时触发。
- UI 层（如 `ScoreBoard`, `GameCanvas`, `Overlay`）在初始化时订阅相关事件，在回调中更新视图。

### 3.4 为什么不使用 Redux/Zustand？

- 游戏渲染依赖 Canvas 绘制，而非模板绑定，状态管理库的优势不明显。
- 单一状态源 + 事件驱动足以满足需求，且零依赖、体积小。
- 若后续扩展多人或复杂 UI，可再引入轻量状态库，但当前无必要。

## 四、与后端交互方式

### 4.1 调用形式

由于“后端”实质是前端模块 `GameCore`，交互为**同步函数调用**，无网络请求。游戏主循环由 `requestAnimationFrame` 驱动，其流程如下：

```javascript
// 主渲染循环（简化）
function gameLoop(timestamp) {
  if (gameCore.state.status === 'PLAYING') {
    if (timestamp - lastTickTime >= gameCore.state.speed) {
      gameCore.tick();      // 后端执行一步逻辑
      lastTickTime = timestamp;
    }
  }
  renderer.draw(gameCore.getState()); // 前端渲染最新状态
  requestAnimationFrame(gameLoop);
}
```

### 4.2 核心调用接口

| 调用方 | 调用目标 | 触发时机 | 返回值/效果 |
|-------|---------|---------|------------|
| `GameContainer.init()` | `gameCore.init(options)` | 页面加载 | 生成初始蛇、食物，状态置为 `IDLE` |
| `InputManager.handleDirection(dir)` | `gameCore.changeDirection(dir)` | 用户输入方向键/滑动 | 若方向合法，更新蛇方向；否则忽略 |
| `InputManager.handlePause()` | `gameCore.pause()` | 空格键或暂停按钮 | 切换 `PLAYING` ↔ `PAUSED` |
| `GameOverModal.restart` | `gameCore.init()` → `gameCore.start()` | 点击“重新开始” | 重置游戏并开始 |
| 渲染循环 | `gameCore.getState()` | 每帧 | 返回当前状态快照（浅拷贝） |

### 4.3 错误处理

- **非法方向**：`changeDirection` 内部检测反向，若违反规则返回 `false`，UI 层忽略该输入。
- **碰撞后游戏结束**：`tick()` 内部调用 `checkCollision()`，若检测到死亡，自动调用 `end()` 并触发 `'gameover'` 事件，UI 层显示结束弹窗。
- **音效播放失败**：音效模块通过 Web Audio API 生成简单音调，若浏览器不支持（极少），静默忽略，不影响游戏。

### 4.4 数据获取策略

- 无 SWR / React Query 等远程数据获取库，因为无网络请求。
- 最高分记录：游戏结束时调用 `localStorage` 写入；游戏初始化时读取，策略为“写时立即同步，读时同步”。

**实现方案**：

- 使用 `window.addEventListener('hashchange', handleRoute)` 监听。
- 路由处理函数中，根据 hash 控制对应覆盖层的显示/隐藏（如 `#/help` 显示 `TutorialOverlay`）。
- 游戏主逻辑始终保持运行，覆盖层不影响 Canvas 渲染（暂停时冻结 Canvas）。

> **注**：若产品要求极致简单，可放弃路由，所有弹窗由游戏内部状态控制。但为未来扩展（如不同游戏入口）保留路由能力，臣认为加入基础 Hash 路由是合理的。

## 六、UI/UX 设计要点

### 6.1 交互规范

| 功能 | 键盘操作 | 触摸操作 | 鼠标操作 |
|------|---------|---------|---------|
| 方向控制 | ↑ ↓ ← → 或 WASD | 在游戏画布上滑动（上下左右） | — |
| 暂停/继续 | 空格键 | 点击暂停按钮 | 点击暂停按钮 |
| 重新开始 | 游戏结束后按 Enter | 点击“重新开始”按钮 | 点击按钮 |
| 关闭操作指引 | 点击“开始游戏”按钮或按 Enter | 点击按钮 | 点击按钮 |

- **方向反冲保护**：输入管理器中，若新方向为当前方向的相反方向（如当前向右，用户按左），直接忽略该输入，不传递给 `GameCore`。
- **触控滑动阈值**：设定最小滑动距离 20px，防止误触；方向判定取滑动向量绝对值较大的轴。

### 6.2 响应式策略

- **画布自适应**：使用 CSS `max-width: 100%; height: auto;` 并监听窗口 `resize` 事件，动态计算画布逻辑像素与物理像素比例。游戏网格大小固定为 20×20，每个格子的大小根据容器宽度/高度等比缩放（取 min 值）。
- **UI 元素布局**：
  - 计分板与控制栏使用 `flex` 布局，在窄屏下垂直排列，宽屏下水平排列。
  - 覆盖层使用 `position: fixed;` + `flex` 居中，背景半透明。
- **断点设计**：
  - 宽度 < 480px：全屏布局，计分板与控制栏缩短间距，按钮尺寸增大（适应触屏）。
  - 宽度 480px~960px：标准布局。
  - 宽度 > 960px：游戏区域居中，两侧留白。

### 6.3 无障碍设计考虑

- **键盘导航**：所有交互按钮均可通过 Tab 键聚焦，Enter/Space 激活。
- **ARIA 标签**：
  - 暂停按钮：`aria-label="暂停游戏"`（状态变化时动态更新）。
  - 游戏结束弹窗：`role="dialog"`、`aria-modal="true"`。
  - 计分板：`aria-live="polite"` 以便屏幕阅读器播报分数变化。
- **对比度**：游戏元素（蛇、食物、背景）色彩对比度不低于 4.5:1（黑白色系下），提供主题切换选项（如高对比度模式）。
- **触控友好**：按钮最小触摸区域 44×44px。

### 6.4 视觉与动画

- **蛇身绘制**：圆角矩形，蛇头略大并用眼睛指示方向，身体渐变色提高辨识度。
- **食物绘制**：闪烁动画（简单缩放呼吸效果），便于玩家定位。
- **暂停状态**：画面半透明灰化，中央显示“⏸ 暂停”。
- **游戏结束**：画面抖动效果（可选），随后显示遮罩和结果。

### 6.5 性能优化

- 渲染循环中使用 `requestAnimationFrame`，仅在状态变更时重绘（非每帧重绘）？但为简化，每帧调用清空重绘，但仅当状态为 `PLAYING` 时执行 `tick`。利用 Canvas 的 `clearRect` 局部重绘（蛇尾和食物变化区域）可进一步优化，但当前方案足以达到 60FPS。
- 避免频繁 DOM 操作：计分板更新使用 `textContent`，不触发重排。
- 音效预生成缓冲：在 `Web Audio` 中创建一次 `AudioBuffer`，后续播放不复创建。

## 五、验收标准

以下条件**全部满足**，方可判定测试通过：

### 5.1 功能验收
1. **启动**：打开页面即自动进入游戏可玩状态（或先显示操作指引，点击后开始）。初始蛇长3、朝右。
2. **方向控制**：键盘方向键和触控滑动均能正确改变蛇方向，且无法反向180°。
3. **碰撞致死**：蛇头撞到网格边界或自身身体任意部分，立即游戏结束（GAME_OVER）。
4. **食物与计分**：蛇头碰到食物后长度+1、得分+1，新食物随机出现在空白格，不得与蛇身重叠。
5. **实时显示**：游戏过程中右上角（或其他固定位置）显示当前得分（和长度可选）。
6. **暂停/继续**：按空格键或点击暂停按钮可暂停/继续；暂停时画面冻结，显示“已暂停”。
7. **游戏结束界面**：半透明遮罩，居中显示“游戏结束”、本局得分、历史最高分、“重新开始”按钮。
8. **重新开始**：点击“重新开始”按钮或游戏结束时自动重置？要求点击后重置。
9. **最高分持久化**：最高分通过 localStorage 本地保存，跨页面访问一致。
10. **操作指引**：首次访问时显示简要操作说明，可关闭；后续访问不再自动显示（除非清除记录）。

### 5.2 性能验收
- 游戏帧率≥30FPS（推荐60FPS），运行稳定无明显掉帧（使用 DevTools Performance 录制10秒钟）。
- 从输入到蛇头转向的延迟≤50ms（可通过手动模拟测量）。
- 内存占用≤50MB，连续运行30分钟无累积增长。

### 5.3 兼容性验收
- 在五种浏览器（Chrome/Firefox/Safari/Edge）最新版上功能正常。
- 在移动设备（iOS Safari / Android Chrome）上触控滑动有效，游戏区域自适应，无布局错乱。
- 屏幕分辨率从320px宽到1920px宽均能完整显示，游戏区域居中。

### 5.4 安全验收

#### 6.3.3 端到端测试（E2E）
- **工具**：Cypress 或 Playwright
- **覆盖场景**：
  - 完整游戏流程（初始化→方向操作→吃食物→撞墙→结束→重新开始）

  - 难度切换
  - 操作指引弹窗
  - 移动端滑动（模拟 touch 事件）
- **关键断言**：
  - 存在指定 DOM 元素（计分板、按钮、覆盖层）
  - 蛇头坐标符合预期（可通过读取 Canvas 像素颜色或暴露内部状态？更佳：暴露游戏状态全局对象用于测试）
  - 页面 url 无变化（单页应用）
- **执行**：`npm run e2e`，可在 CI 中使用无头模式。

### 6.4 持续集成中的测试流程

采用 GitHub Actions（或类似 CI 工具），配置如下流水线：

```yaml
name: Snake Game CI

jobs:
```

- 若单元测试失败，阻断后续流水线。

- 性能检测结果作为参考，不强制阻断但记录到日志。
