# 原始需求

做一个贪吃蛇游戏

# pm

## 一、背景与目标

**背景**：贪吃蛇游戏是一款经典的休闲益智游戏，玩家通过操控一条不断移动的蛇，在限定区域内吃食物并避免撞墙或撞到自己，旨在重温经典玩法，为玩家提供轻松愉快的娱乐体验。

**核心问题**：缺乏一款稳定、可玩性高的贪吃蛇游戏，玩家希望能在移动设备或桌面端随时游玩。

**预期目标**：开发一款功能完整、操作流畅的贪吃蛇游戏，支持单人模式，包含基本游戏规则、分数记录、难度递增等核心功能，确保玩家获得沉浸式体验。

## 二、用户故事

1. **作为玩家**，我希望通过键盘（桌面端）或屏幕滑动（移动端）控制蛇的移动方向，以便灵活地操控蛇去吃食物而不撞墙或撞到自己。
2. **作为玩家**，我希望蛇每吃掉一个食物，身体就增长一个单位，同时得分增加，以便获得持续挑战和成就感。
3. **作为玩家**，我希望当蛇撞到墙壁或自己的身体时，游戏结束并显示本局得分，以便了解自己的表现并决定是否重新开始。

## 三、功能列表

| 功能模块 | 详细描述 | 优先级 |
|---------|---------|--------|
| 游戏界面 | 显示固定大小的游戏区域（例如20x20网格）、蛇身、食物、当前得分 | P0 |
| 蛇的移动控制 | 通过方向键或滑动控制上下左右，蛇头方向改变后，身体按原路径跟随 | P0 |
| 食物生成 | 在游戏区域内随机生成一个食物，不与蛇身重叠 | P0 |
| 吃食物与增长 | 蛇头接触到食物后，食物消失，蛇身增加一节，得分+1 | P0 |
| 碰撞检测与游戏结束 | 蛇头撞到墙壁或蛇身时，游戏结束，显示最终得分和重新开始按钮 | P0 |
| 重新开始 | 游戏结束后，玩家可点击“重新开始”按钮重置游戏状态 | P0 |
| 暂停/继续 | 游戏进行中，玩家可按键或点击暂停，再次点击继续 | P1 |
| 分数记录 | 本地保存最高分，并在游戏界面上显示 | P1 |
| 难度递增 | 每吃5个食物，蛇的移动速度略微增加 | P1 |
| 游戏音效 | 吃食物、游戏结束等场景播放简短的音效 | P2 |

## 四、非功能需求

- **性能**：游戏帧率不低于30fps，移动端滑动操作响应延迟不超过100ms。
- **兼容性**：支持主流浏览器（Chrome、Firefox、Safari、Edge）最新版本，移动端适配横竖屏。
- **可用性**：游戏区域清晰可见，蛇身颜色与背景对比明显，食物易于辨认；控制按钮在移动端需有触感反馈。
- **可维护性**：代码采用模块化设计，游戏逻辑与渲染分离，便于后续扩展新功能（如双人模式、皮肤系统）。
- **离线支持**：游戏资源（HTML、CSS、JS、音效）可打包为单页面应用，支持无网络环境下运行。

## 五、验收标准

1. **核心玩法**：玩家可通过键盘或滑动控制蛇移动，吃到食物后蛇身增长、得分增加，撞墙或撞自身后游戏结束并显示分数。
2. **完整交互**：游戏初始状态显示“开始游戏”按钮或自动开始；游戏过程中可暂停/继续；游戏结束后可点击“重新开始”回到初始状态。
3. **分数记录**：最高分在本地持久化存储，每次游戏结束后更新，并在游戏界面右上角显示。
4. **性能稳定**：在标准配置桌面设备和主流智能手机上，游戏运行流畅，无卡顿、崩溃现象。
5. **界面美观**：游戏界面配色协调，蛇身、食物、网格线具有良好视觉区分度，字体清晰可读。
6. **自适应屏幕**：游戏区域在不同分辨率下保持比例，移动端触摸控件位置合理，防误触。

# 系统架构设计方案

## 一、技术选型

| 领域 | 选型 | 选择理由 |
|------|------|----------|
| 前端框架 | 原生 JavaScript (ES6+) + HTML5 Canvas | 游戏逻辑简单，无需重量级框架，原生 API 可直接操控像素级渲染，性能最佳，且满足离线单页应用需求 |
| 语言增强 | TypeScript（可选） | 提供类型检查，提升大型代码库的可维护性，但本游戏规模较小，可采用 JSDoc 或直接使用 JS |
| 游戏渲染 | Canvas 2D | 支持高帧率、像素级控制，适合网格类游戏，比 DOM 操作更高效 |
| 样式与布局 | CSS3 Flexbox + Grid | 确保游戏区域自适应各种屏幕尺寸，移动端横竖屏适配 |
| 本地存储 | localStorage | 持久化存储最高分记录，无需后端数据库 |
| 音效 | Web Audio API 或预加载 Audio 元素 | 轻量音效播放，无需外部音频库，支持短促的吃食物、结束音效 |
| 模块打包 | Vite 或 Webpack（可选单 HTML 文件时无需打包） | 若项目需要分模块开发，可用 Vite 快速开发，最终可产出单一 HTML/JS/CSS 文件 |
| 部署 | 静态文件托管（如 GitHub Pages、Netlify、或任何 Web 服务器） | 零服务器成本，支持离线运行（全资源打包为单文件） |

**核心设计原则**：游戏逻辑与渲染完全分离，输入处理独立，状态管理通过一个中央 GameState 对象完成。

---

## 二、模块划分

整个系统分为以下模块（对应独立的 JS 文件或 ES6 Module）：

| 模块 | 职责边界 |
|------|----------|
| **GameEngine** | 游戏主循环管理（requestAnimationFrame）、帧率控制、状态机（等待、进行中、暂停、结束） |
| **GameConfig** | 游戏配置常量：网格大小（20×20）、初始速度、缩放倍数、颜色方案等 |
| **GameState** | 单一数据源：蛇的位置数组、当前方向、食物位置、得分、最高分、难度等级（速度倍数） |
| **SnakeController** | 蛇的移动逻辑：根据方向更新蛇头坐标，身体跟随（移除尾端，头部插入），碰撞检测入口 |
| **FoodManager** | 食物随机生成逻辑：保证不落在蛇身上，提供食物坐标 |
| **CollisionDetector** | 碰撞检测：蛇头是否超出边界，是否与自身重叠；返回布尔值 |
| **InputHandler** | 键盘事件监听（方向键、空格暂停）、移动端触摸滑动识别（滑动方向映射上下左右）；防抖过滤快速连续同方向输入 |
| **Renderer** | Canvas 渲染：清屏、绘制网格、蛇身（分段颜色渐变或纯色）、食物、得分面板；支持屏幕自适应缩放 |
| **ScoreManager** | localStgorage 读写最高分，提供当前得分、最高分接口 |
| **AudioManager** | 预加载音效文件（吃食物、游戏结束），提供播放方法 |


模块依赖关系（简化）：

```
InputHandler → GameEngine → GameState (更新)
                         ↕
              SnakeController / FoodManager / CollisionDetector
                         ↓
                     Renderer
                         ↓
              ScoreManager (读/写 localStorage)
              AudioManager (播放音效)
```

## 三、数据流图（文字描述）

### 核心用户操作 → 响应链路

1. **初始化**：GameEngine 读取 GameConfig，创建 GameState（蛇初始位置在网格中央，方向向右，得分 0），调用 Renderer 绘制初始界面，显示“按空格开始”。
2. **开始游戏**：用户按空格或点击开始按钮 → InputHandler 捕获 → GameEngine 状态变为“进行中”，启动主循环。
3. **每次帧循环**：
   - GameEngine 根据当前速度间隔（由难度调整）决定是否更新逻辑帧。
   - 若更新：先由 SnakeController 计算基于当前方向的新蛇头坐标。
   - 然后 CollisionDetector 检查新蛇头是否撞墙/撞自身。
     - 若碰撞 → GameState 状态变为“结束”，Renderer 绘制 Game Over 画面，ScoreManager 更新最高分，AudioManager 播放结束音效。
     - 若无碰撞 → 检查新蛇头是否与食物位置重合。
       - 若吃到食物：调用 FoodManager 生成新食物，SnakeController 增加身体（不删除尾部），GameState 得分 +1，AudioManager 播放吃食物音效。若得分达到 5 的倍数，GameState 更新难度（速度增加）。
       - 若未吃到：SnakeController 正常移动（头部插入，尾部移除）。
     - 更新 GameState.snake 和 GameState.food。
   - Renderer 根据最新 GameState 重新绘制整个画面（含网格、蛇、食物、分数面板）。
4. **暂停**：用户按空格或点击暂停按钮 → InputHandler 触发 → GameEngine 切换暂停状态，不执行逻辑更新，Renderer 可绘制暂停遮罩。
5. **重新开始**：游戏结束后用户点击“重新开始” → GameEngine 重置 GameState 到初始状态，清空画布，回到步骤 2 等待开始。

### 移动端触摸数据流

触摸事件 → InputHandler 记录 touchstart 和 touchend 坐标差，计算方向（上下左右阈值判断）→ 更新 GameState.direction → 与键盘流程合并。

## 四、接口设计概要

本游戏为纯前端应用，无服务器端 API。所有“接口”指内部模块的方法或事件。

| “端点”/接口方法 | 用途 | 参数/返回值 |
|----------------|------|------------|
| `GameEngine.start()` | 启动/恢复游戏主循环 | 无 |
| `GameEngine.pause()` | 暂停游戏 | 无 |
| `GameEngine.reset()` | 重置游戏状态至初始 | 无 |
| `GameEngine.setDirection(direction)` | 设置蛇移动方向（由 InputHandler 调用） | `direction`: 'up'\|'down'\|'left'\|'right' |
| `SnakeController.move(direction, isGrowing)` | 执行一步移动，返回新蛇头坐标及碰撞信息 | 内部方法 |
| `CollisionDetector.checkWall(head, gridWidth, gridHeight)` | 检查蛇头是否撞墙 | 返回 `boolean` |
| `CollisionDetector.checkSelf(head, snakeBody)` | 检查蛇头是否撞自身 | 返回 `boolean` |
| `FoodManager.generate(snakeBody, gridWidth, gridHeight)` | 生成新的食物坐标 | 返回 `{x, y}` |
| `Renderer.render(state)` | 根据 GameState 绘制一帧 | 无返回值 |
| `ScoreManager.getHighScore()` | 从 localStorage 读取最高分 | 返回 `number` |
| `ScoreManager.saveHighScore(score)` | 保存最高分到 localStorage | 无返回 |
| `AudioManager.playEat()` / `playGameOver()` | 播放对应音效 | 无 |

无网络 RPC 或 HTTP 接口，游戏完全本地运行。

## 五、部署架构

采用 **单页面应用 (SPA)** 模式，所有资源打包为一个 HTML 文件（或一个 HTML + 一个 JS + 一个 CSS + 音效文件），静态部署。

```
┌──────────────────────────────────┐
│          CDN / Web Server        │
│ (GitHub Pages / Netlify / Nginx) │
├──────────────────────────────────┤
│  index.html (含内联 CSS/JS 或引用)│
│  sound_eat.mp3 / sound_end.mp3   │
└──────────────────────────────────┘
         ↑ 用户浏览器访问
```

**环境要求**：
- 浏览器：支持 ECMAScript 2015+ 和 Canvas 2D 的现代浏览器（Chrome/Firefox/Safari/Edge 最新版）。
- 无后端依赖，完全离线运行（将音效作为 Data URI 内联到 HTML 中可做到完全离线单文件）。
- 推荐使用 `navigator.serviceWorker` 缓存资源实现离线可用（可选增强）。

**扩展考虑**：
- 若未来需多人对战，可引入 WebSocket 服务器 + Node.js 后端，此时架构可转为前端渲染 + 后端游戏逻辑，本设计仍可保留客户端渲染和输入部分。

## 六、风险与权衡

| 风险 | 描述 | 应对策略 |
|------|------|----------|
| **移动端触摸精度** | 玩家滑动可能被误识别为错误方向，或滑动距离过短导致无响应 | 在 InputHandler 中设置最小滑动距离阈值（如 20px），且忽略对角线（只取水平/垂直分量最大的方向）；提供虚拟方向按钮作为备选方案（需求中 P2 但可降级） |
| **游戏帧率不足** | 在低端设备上 Canvas 全网格重绘可能导致掉帧 | 采用双缓冲（直接使用 Canvas 缓冲）和局部更新优化：仅重绘变化的网格单元（蛇头、蛇尾、食物），减少绘制量；保持 requestAnimationFrame 调用 |
| **速度递增导致难度失控** | 当蛇身很长时，移动速度过快可能使玩家无法反应 | 设置速度递增上限（例如初始 150ms/步，每次增加 10ms，最低 60ms/步），并在 UI 显示当前难度级别 |
| **音效加载延迟** | 首次点击时音效文件可能尚未加载完成 | 在页面加载时预先通过 Audio() 构造函数加载所有音效（或使用 Web Audio 解码）；若加载失败静默降级（无音效不影响游戏） |
| **localStorage 被禁用** | 隐私模式下某些浏览器可能提示用户不允许写入 | 使用 try-catch 包裹 localStorage 操作，若禁用则最高分仅在会话内保存（不持久化） |
| **屏幕适配变形** | 在非常规宽高比的屏幕上游戏区域可能被拉伸 | 游戏网格渲染区域采用固定逻辑尺寸（例如 400x400 像素），通过 CSS 实现等比例缩放（max-width + aspect-ratio），Canvas 内部始终以 20x20 格子计算，保留像素无关性 |
| **键盘冲突** | 玩家快速连续按方向键导致蛇在极短时间内反向移动（撞到自己） | InputHandler 实现方向队列或禁止反向输入（当前方向与即将设置的方向相反时忽略） |

**设计权衡**：
- **模块化 vs 简单性**：本设计采用较细的模块划分（7个模块），虽然增加文件数，但便于单元测试和后续功能扩展（如双人模式可复用 SnakeController 和 CollisionDetector，只需增加一个 GameState）。
- **Canvas vs SVG**：Canvas 更适合频繁重绘的动画，SVG 更适合静态图表，故选择 Canvas。
- **无框架 vs 框架**：原生 JS 避免引入框架体积，性能可控，代码量不大，且单页面打包更简单；若后续需复杂 UI 交互（如设置界面），可考虑轻量框架如 Preact，但当前需求无需。

## 二、API 接口详细设计

所有接口均走 HTTP，请求/响应格式为 JSON，基础路径 `/api/v1`。

### 2.1 提交分数

- **HTTP 方法与路径**：`POST /api/v1/scores`
- **请求 Body**：
```json
{
  "playerName": "无敌暴龙战士",
  "score": 158,
  "snakeLength": 22,
  "playDuration": 345
}
```
- **响应格式**：
```json
{
  "code": 200,
  "message": "OK",
  "data": {
    "id": 1024,
    "rank": 3
  }
}
```
- **状态码**：
  - `200`：成功
  - `400`：参数校验失败（如分数为负数、昵称过长）
  - `409`：昵称已存在且新分数低于历史最高分（可配置仅允许更新更高分）
- **说明**：仅当新分数高于该玩家历史最高分时更新，否则返回 `409` 并提示“历史最高分已更高”。

### 2.2 获取排行榜

- **HTTP 方法与路径**：`GET /api/v1/scores/top?limit=10&offset=0`
- **请求参数**：
  - `limit`（可选，默认10，最大100）
  - `offset`（可选，默认0）
- **响应格式**：
```json
{
  "code": 200,
  "data": {
    "total": 1024,
    "items": [
      {
        "rank": 1,
        "playerName": "独孤求败",
        "score": 9999,
        "snakeLength": 200,
        "createdAt": "2025-04-01T10:00:00Z"
      }
    ]
  }
}
```
- **状态码**：`200`
- **说明**：按分数降序排列，返回指定范围的排行。

### 2.3 查询玩家历史记录

- **HTTP 方法与路径**：`GET /api/v1/scores/:playerName`
- **响应格式**：
```json
{
  "code": 200,
  "data": {
    "playerName": "清风",
    "maxScore": 850,
    "gameCount": 23,
    "totalPlayDuration": 7200
  }
}
```
- **状态码**：`200` / `404`

### 2.4 健康检查

- **HTTP 方法与路径**：`GET /api/v1/health`
- **响应**：`{ "status": "ok", "timestamp": "..." }`
- **状态码**：`200`

## 三、核心业务逻辑

### 3.1 分数提交流程（伪代码）

```javascript
function submitScore(playerName, score, snakeLength, playDuration) {
    // 1. 参数清洗与校验
    if (!playerName || playerName.length > 32) return error(400, '昵称无效');
    if (score < 0 || snakeLength < 0 || playDuration < 0) return error(400, '参数异常');

    // 2. 查询玩家记录
    const existing = db.query('SELECT * FROM scores WHERE playerName = ?', [playerName]);
    
    if (existing) {
        // 3. 若已有记录，仅当新分数更高时更新
        if (score > existing.score) {
            db.update('scores', { score, snakeLength, playDuration, createdAt: now() }, { id: existing.id });
        } else {
            // 返回409（可选携带历史最高分）
            return error(409, '恭喜已达历史最高分：' + existing.score);
        }
    } else {
        // 4. 插入新记录
        db.insert('scores', { playerName, score, snakeLength, playDuration, createdAt: now() });
    }

    // 5. 计算新排名
    const rank = db.query('SELECT COUNT(*) + 1 FROM scores WHERE score > ?', [score]);

    return success({ id, rank });
}
```

### 3.2 防刷分措施

- 同一 `playerName` 每30秒内仅允许提交一次（内存缓存 + 滑动窗口）。
- 分数上限限制：单局最高分不超过网格总数（20×20=400），防止客户端伪造。
- 对 `playDuration` 与分数做合理性校验（如100分时游戏时长不低于60秒，否则疑似脚本刷分）。

## 四、数据库设计

### 4.1 表结构（SQLite DDL）

```sql
CREATE TABLE IF NOT EXISTS scores (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    player_name VARCHAR(32) NOT NULL UNIQUE,
    score INTEGER NOT NULL DEFAULT 0 CHECK(score >= 0),
    snake_length INTEGER DEFAULT 0,
    play_duration INTEGER DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP

```

### 4.2 索引策略

- 主键索引（自动）
- `score` 字段降序索引（加速排行榜查询）：`CREATE INDEX idx_score_desc ON scores(score DESC);`
- `player_name` 唯一性约束已隐含索引

### 4.3 数据迁移方案

- 采用 `CREATE TABLE IF NOT EXISTS` 或版本管理表（如 `schema_version`）。
- 初始版本 `v1` 即当前表结构。后续如需增加字段（如 `avatar_url`），使用 `ALTER TABLE` 加字段。

## 五、错误处理与安全

### 5.1 错误处理策略

- 统一错误响应格式：`{ code, message, details? }`
- 所有未知异常返回 `500`，并记录服务器日志（日志文件 stdout/stderr）。
- 对客户端参数错误返回 `4xx`，明确错误原因。

### 5.2 鉴权方案

- 当前为公开排行榜，无需用户注册，但需防范恶意刷分。
- 后续若需防刷，可引入简单的 `X-Player-Token`（基于玩家名+设备指纹的HMAC签名），服务端校验有效性。

### 5.3 数据校验规则

- 服务端双重校验所有输入：
  - 昵称：长度1~32字符，只允许字母、数字、中文、下划线、短横线，正则 `^[\u4e00-\u9fa5a-zA-Z0-9_-]{1,32}$`
  - 分数：非负整数，上限 400。
  - 蛇身长度：非负整数，上限 400。
  - 游戏时长：非负整数（秒），参考上限 3600（1小时）。

### 5.4 安全建议

- 使用 HTTPS 部署，防止中间人攻击篡改分数。
- 对 `@`、`%`、`<script>` 等危险字符做转义或拒绝，防止 XSS 或 SQL 注入（参数化查询已免疫）。
- 添加请求频率限制（如每IP 10次/分钟）。

# frontend

# 贪吃蛇游戏前端设计方案

## 一、页面结构设计

整个游戏以单页面应用（SPA）形式呈现，无需传统多页路由。游戏体验分为三个清晰的**视图状态**，通过状态机切换，仅在Canvas视口中绘制不同内容。

| 视图名称 | 触发条件 | 主要内容 |
|---------|---------|---------|
| **开始视图** | 页面加载/游戏结束后的“重新开始” | 游戏标题、最高分显示、操作说明（键盘/触控）、**开始游戏**按钮 |
| **游戏视图** | 玩家点击“开始”或按空格键 | 全屏游戏区域（Canvas）、实时得分、当前速度等级、暂停/继续按钮 |
| **结束视图** | 蛇撞墙或撞自身 | 游戏结束面板：显示本局得分、最高分、蛇身长度、重新开始按钮（可直接开始新游戏） |

**页面布局**（桌面端与移动端通用）：
- 顶部：分数面板（当前得分、最高分、速度等级）
- 中部：游戏Canvas区域（自适应比例，保持正方形）
- 底部（移动端）：方向控制按钮（上下左右四个半透明按钮，可选显示）

**响应式策略**：游戏区域始终为正方形，宽度占视口短边的80%（限高保证完整显示），移动端竖屏时方向按钮位于底部。

## 二、组件树

由于采用原生JavaScript模块化开发，组件以**对象/类**形式组织，不存在框架组件树。但从设计角度看，可将UI划分为以下逻辑组件：

```
App（入口）
├── GameContainer（游戏容器，负责整体布局）
│   ├── ScoreBoard（得分面板）
│   │   ├── CurrentScore（当前得分）
│   │   ├── HighScore（本地最高分）
│   │   └── SpeedLevel（当前速度等级）
│   ├── Canvas（游戏画布，核心渲染区域）
│   ├── Overlay（覆盖层，用于显示开始/结束/暂停面板）
│   │   ├── StartPanel（开始面板）
│   │   ├── PausePanel（暂停面板）
│   │   └── GameOverPanel（结束面板）
│   └── MobileControls（移动端触控按钮，可选）
│       ├── UpButton
│       ├── DownButton
│       ├── LeftButton
│       └── RightButton
└── AudioControl（音效开关，悬浮在角落）
```

**职责说明**：
- `GameContainer`：管理所有DOM元素创建、布局、自适应缩放（监听resize事件）
- `ScoreBoard`：纯显示组件，从GameState读取数据，实时更新DOM
- `Canvas`：接收Renderer模块绘制指令，暴露canvas元素引用
- `Overlay`：用于显示非游戏时的模态面板，通过CSS类控制显示/隐藏
- `MobileControls`：仅移动端显示，通过`touchstart`事件回调触发方向改变
- `AudioControl`：提供音效播放接口，并支持静音切换（localStorage持久化）

## 三、状态管理方案

### 3.1 方案选择
采用**单一游戏状态对象（GameState）** 模式，不引入第三方状态管理库。原因：游戏状态简单且集中，单个对象足以管理所有可变数据，配合发布-订阅模式实现视图更新。

### 3.2 核心状态结构
```javascript
class GameState {
  // 游戏模式状态
  gameStatus: 'idle' | 'playing' | 'paused' | 'gameover';
  
  // 蛇相关
  snake: [{x, y}, ...];   // 蛇身坐标数组（蛇头为第一个元素）
  direction: 'UP' | 'DOWN' | 'LEFT' | 'RIGHT';
  nextDirection: ...;     // 缓冲方向，用于处理同帧多次输入
  growPending: boolean;   // 是否正在等待增长（吃食物后下一帧长度+1）
  
  // 食物
  food: {x, y};
  
  // 计分与难度
  score: number;
  highScore: number;      // 从localStorage读取
  speedLevel: number;     // 当前速度等级（0: 基础速度）
  totalEaten: number;     // 总吃食物数（用于判定速度提升）
  
  // 时间
  lastMoveTime: number;   // 上次逻辑移动时间戳（ms）
  moveInterval: number;   // 当前移动间隔（ms），随speedLevel减小
}
```

### 3.3 状态变更流程
所有状态变更通过`GameEngine`调用具体模块（`SnakeController`、`FoodManager`、`CollisionDetector`）后，统一更新GameState，然后触发`Renderer.render()`进行绘制。采用**观察者模式**，GameState通过`eventBus`（简单自定义事件）通知订阅者（ScoreBoard、Renderer等）。

## 四、与后端交互方式

### 4.1 交互策略
当前版本暂不调用后端API，所有数据（最高分）存储在`localStorage`。但为了未来扩展，设计**抽象服务层** `ScoreService`：

```javascript
// ScoreService.js
const ScoreService = {
  // 本地存储实现
  localStorage: {
    getHighScore() { return localStorage.getItem('snake_highscore') || 0; },
    setHighScore(score) { localStorage.setItem('snake_highscore', score); }
  },
  
  // 远端API实现（供未来替换）
  remote: {
    async submitScore(playerName, score) { ... },
    async getTopScores(limit) { ... },
  },
  
  // 当前活跃策略（可通过特性开关切换）
  current: null, // 初始化时设置为 localStorage
  
  init() {
    this.current = this.localStorage; // 默认本地
  },
  
  getHighScore() { return this.current.getHighScore(); },
  setHighScore(score) { this.current.setHighScore(score); }
};
```

### 4.2 数据获取策略
- **最高分**：应用启动时从`ScoreService.getHighScore()`同步获取，显示在ScoreBoard
- **游戏结束**：当新分数高于历史最高分时，调用`ScoreService.setHighScore()`更新localStorage
- **未来扩展**：若开启远程排行榜，可在游戏结束时异步提交分数，并显示“提交中/成功/失败”反馈；排行榜数据使用`React Query`风格的缓存策略（手动刷新，缓存10分钟）。

### 4.3 错误处理
- 本地存储：`try-catch`包裹`localStorage`操作，若失败（隐私模式可能抛出异常），降级为内存存储，不影响游戏进行。
- 远端API调用：捕获异常后静默失败，不影响用户体验，仅在console输出警告。若需要，可增加重试机制（最多3次，指数退避）。

### 4.4 离线支持
所有资源（HTML、CSS、JS、音效文件）打包为单HTML文件，使用`<script>`内联，音效Base64编码。游戏运行完全离线。

## 五、页面路由设计

此游戏为单页应用，无需传统路由。但为清晰化视图切换，设计内部路由对象：

| 路径（状态名） | 对应视图 | 是否鉴权 | 说明 |
|---------------|---------|---------|------|
| `#/start` | 开始视图 | 否 | 默认路由，显示开始按钮 |
| `#/game` | 游戏视图 | 否 | 主游戏进行中 |
| `#/paused` | 暂停视图 | 否 | 游戏暂停时覆盖层 |
| `#/gameover` | 结束视图 | 否 | 游戏结束后显示 |

通过监听`hashchange`事件或手动调用`navigateTo`方法切换视图。实际上仅用于单页内部状态记录，支持浏览器前进后退（例如玩家想回退到开始画面）。路由变化触发`GameEngine.setState()`，并结合`Overlay`组件显示对应面板。

## 六、UI/UX 设计要点

### 6.1 交互规范
- **桌面端键盘控制**：
  - ↑↓←→ 或 W/A/S/D 控制方向
  - 空格键：开始游戏/暂停/继续
  - ESC键：游戏中退回开始画面（需确认弹窗？可设计为直接退出并显示游戏结束面板）
- **移动端触控**：
  - 在游戏区域滑动（任意方向）触发方向变化，滑动距离大于20px生效，避免误触
  - 提供固定的四方向按钮（半透明，圆角，1.5rem大小，位于画面底部中央）
  - 双击或点击暂停按钮暂停；游戏结束面板点击“重新开始”
- **输入防抖**：每帧只允许一次方向变化，防止蛇在极短时间内连续回头导致撞身（通过缓冲队列处理）。

### 6.2 响应式策略
- **游戏区域**：始终为正方形，宽度 = `Math.min(window.innerWidth * 0.9, window.innerHeight * 0.8, 600px)`，高度与宽度相同，居中显示。
- **移动端方向按钮**：仅当`window.innerWidth < 768px`或`'ontouchstart' in window`时显示，否则隐藏。
- **字体大小**：得分面板字体使用`clamp(1rem, 3vw, 1.5rem)`。
- **横竖屏适配**：监听`orientationchange`事件，重新计算游戏区域尺寸，并调整方向按钮位置（横屏时可移至右侧）。

### 6.3 无障碍设计考虑
- 所有按钮和交互元素需具有可访问性标签（`aria-label`），例如“向上方向按钮”、“开始游戏按钮”
- 得分和游戏状态使用`aria-live="polite"`区域动态播报（通过屏幕阅读器）
- 游戏结束面板聚焦到“重新开始”按钮（使用`autofocus`）
- 颜色对比度：蛇身使用亮色（#4CAF50绿色），食物使用红色（#FF5722），背景黑色（#1a1a2e），网格线浅灰（#333），确保文字和背景对比度≥4.5:1
- 支持关闭动画（`prefers-reduced-motion`媒体查询），减少闪烁和移动效果

### 6.4 视觉风格
- 采用**深色主题**（暗蓝紫背景 + 霓虹绿蛇身 + 橙色食物），符合游戏氛围
- 蛇身分段渐变（从头部到尾部由亮绿渐变为深绿）
- 食物旋转动画（CSS spin），吃食物时粒子扩散效果（Canvas绘制）
- 分数增加时数字跳动效果（CSS transform scale）

### 6.5 性能优化
- 使用`requestAnimationFrame`驱动渲染循环，但逻辑更新使用`setTimeout`控制速度（避免CPU占用过高）
- 只在蛇移动后请求动画帧（节省电量），但需保持至少30fps的背景刷新（如食物闪烁）
- Canvas离屏缓冲：预绘制静态背景（网格、边界），避免每帧重绘
- 移动端避免使用`touchmove`连续触发，改用`touchend`结合坐标计算方向

## 四、边界与异常测试

### 4.1 边界值测试

| 测试场景 | 输入/操作 | 预期结果 |
|----------|-----------|----------|
| **网格边界碰撞**：蛇头在(0,0)且方向为UP或LEFT | 按上键或左键 | 蛇头即将出界，游戏结束（碰撞检测正确） |
| **网格边界碰撞**：蛇头在(19,19)且方向为DOWN或RIGHT | 按下键或右键 | 同上 |
| **蛇身占满整个网格**（20x20=400节） | 通过调试设置蛇身覆盖所有格子 | 食物无法生成（应进入胜利状态或无限循环，按设计无处理，需确认是否处理） | 说明：当前设计无“胜利”条件，但理论上可行，需确保不崩溃 |
| **单次吃多个食物**（食物生成在第一步碰撞点） | 连续快速移动使蛇头瞬间经过两个食物（极低概率） | 按设计一次只能吃一个，新食物在下一帧生成，可接受 |
| **速度等级上限**：连续吃食物达到最高速度 | 吃50个食物，速度等级=10（每次吃5个升一级） | 速度达到最快后不再增加，游戏仍稳定运行（帧率不低于30fps） |
| **最高分溢出**：localStorage中存入超大数值 | 手动设置highScore为1000000 | 游戏正常显示，无数字格式问题（建议限制最高分显示宽度） |

### 4.2 异常输入测试

| 测试场景 | 输入/操作 | 预期结果 |
|----------|-----------|----------|
| **快速连续按键**（同帧内多次方向改变） | 在极短时间内按上、左、右三次 | 仅最后一个有效方向生效，蛇按该方向移动（InputHandler需防抖过滤同帧重复） |
| **反向移动**：蛇向右时按左键 | 按左键 | 无效，蛇继续向右（不能原地逆行） |
| **移动端滑动方向模糊**：滑动角度45度 | 从下方向右上方滑动 | InputHandler应识别为主要方向（基于滑动起点与终点角度）或保持原方向 |
| **移动端多点触控**：同时触摸两个方向按钮 | 同时按左键和上键 | 按设计，每个按钮独立事件，可能造成冲突；建议取最后触摸的方向 |
| **浏览器窗口缩放** | 手动缩放页面到150%或50% | 游戏区域保持正方形比例，Canvas自适应，不出现黑边或截断 |
| **音效文件加载失败** | 将音效文件删除或网络中断 | 游戏正常进行，无音效播放，控制台输出错误日志，不弹出全局错误 |
| **localStorage不可用**（隐私模式或禁用） | 在无痕模式下运行游戏 | 游戏正常进行，分数记录功能降级（显示0且不保存），不崩溃 |
| **后端API超时/500错误** | 提交分数时后端无响应 | 前端应有友好提示（如“排行榜暂时不可用”），游戏体验不受影响 |

## 五、验收标准

> 以下条件全部满足方可判定测试通过，准予上线。

1. **核心玩法完整**：玩家可通过键盘或触摸滑动控制蛇移动，吃到食物后蛇身增长、得分增加，撞墙或撞自身后游戏结束并显示得分与重新开始按钮。**（P0通过，P0全部测试用例通过）**

2. **全部P0测试用例通过率100%**，P1通过率≥95%，P2通过率≥80%（可根据迭代调整）。

3. **性能达标**：
   - 在桌面端（Chrome, 2K屏）运行游戏，帧率始终≥30fps（使用`requestAnimationFrame`回调次数监控）。
   - 移动端（iPhone 12或等性能机型）触摸滑动延迟≤100ms（通过手动测试感知无明显滞后）。
   - 游戏运行30分钟后内存占用增长不超过初始状态的20%。

4. **兼容性**：
   - 在Chrome 120+、Firefox 120+、Safari 15+、Edge 120+四个浏览器最新版本上核心功能（P0）全部通过。
   - 移动端适配横竖屏转换：切换屏幕方向后游戏区域自动重绘，保持正方形比例，无错位。

5. **离线可用**：
   - 将游戏资源打包为单个HTML文件，在没有网络的环境（断网）下打开，所有功能正常运行（音效需要预加载为base64内联或使用Web Audio避免加载问题）。

6. **后端API（如果部署）**：
   - 所有接口按规范响应，状态码正确，参数校验覆盖边界（空值、负数、过长昵称等）。
   - 排行榜数据按分数降序排列正确，分页正常。

7. **安全无重大缺陷**：不存在可能导致游戏崩溃、浏览器崩溃、或数据泄露的漏洞。无XSS注入（尤其玩家昵称不可执行脚本）。

8. **自描述性**：游戏界面有清晰的文字提示（操作说明、暂停面板、结束面板），移动端方向按钮有触感反馈（颜色变化或轻微动画）。

| 变更类型 | 必须回归的测试模块 |
|----------|-------------------|
| 蛇移动逻辑修改 | TC-001~TC-006（全部方向与碰撞）、TC-007（暂停）、TC-009（难度） |
| 食物生成算法修改 | TC-004、TC-005（增加对蛇重叠的回归） |
| Canvas渲染变更 | 所有与显示相关的用例（TC-001~TC-012）、边界测试中的渲染检查 |
| 触摸控制优化 | TC-003（移动端模拟）、边界测试中滑动方向识别 |
| 后端API变更 | API-013~API-016，以及异常测试中的网络异常场景 |
| 音效/静音功能 | TC-011~TC-012 |
| 打包流程变更 | 离线测试（断网运行） |
| 任何变更 | 至少运行P0核心测试用例（TC-001~TC-008） |

### 6.2 自动化测试策略

| 测试分层 | 工具/框架 | 覆盖范围 |
|----------|-----------|----------|
| **单元测试** | Vitest（或Jest） | SnakeController（移动、增长）、CollisionDetector（边界、自身）、FoodManager（生成不重叠）、ScoreManager（localStorage读写）、GameState（状态切换）。预计20个单元测试。 |
| **组件测试** | 使用JSDOM + Canvas mock | 与渲染无关的逻辑模块：InputHandler（键盘事件解析）、GameEngine（帧循环控制） |
| **集成测试** | Puppeteer（无头浏览器） | 自动化打开游戏页面，模拟键盘输入，截图比对关键帧（如初始界面、吃食物后、游戏结束）。可验证状态机切换。 |
| **API自动化测试** | Supertest（Node）或Postman Newman | 后端API功能与边界测试，自动运行全部API用例，校验状态码与响应结构。 |
| **性能测试** | Lighthouse（Performance Audit） | 获取FPS、加载性能等指标。 |
| **兼容性测试** | BrowserStack（手动+自动截图） | 在多个浏览器/设备上运行核心用例，进行视觉比对。 |

**CI集成流程**：
1. 每次`git push`或`pull request`触发GitHub Actions流水线。
2. 运行单元测试 + 集成测试（Puppeteer headless），失败则阻止合并。
3. 若包含后端代码，运行API测试（使用内存数据库）。
4. 合并到`main`分支后，自动部署到测试环境，并触发性能测试与多浏览器截图。
5. 人工验收后，手动触发回归测试（全量手动用例）然后发布。

### 6.3 持续集成中的测试流程

```

    ↓ 通过
```
